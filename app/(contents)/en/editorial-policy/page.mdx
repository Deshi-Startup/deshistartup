# Editorial Policy

Deshi Startup earns trust through verified sources, transparent history, and fast corrections. We borrow from Wikipedia's verifiability model, but keep the workflow small-team friendly.

## Core Rules

1. **Verifiability first.** Readers should be able to open the source behind a claim.
2. **Official sources first.** Use official or primary sources for law, tax, registration, payment, and government programme content.
3. **No source, no high-risk claim.** High-risk claims should not be published without sources.
4. **Neutral language.** Avoid promotional claims about organizations, people, tools, or programmes.
5. **Correct quickly.** Serious reports should be triaged quickly, then corrected or marked with a warning when needed.

## Risk Tiers

| Risk | Examples | Publish rule |
|---|---|---|
| Low | spelling, flow, founder mindset, general checklists | one editor review |
| Medium | market data, ecosystem lists, funding/programme details | source required; update when outdated |
| High | legal, tax, VAT, RJSC, employment, fintech, healthtech, privacy, import/export, government incentives | official/reputable source, `last_verified`, and editor/expert signoff |

## Review Workflow

1. **Pending edit:** a user clicks **Edit this page** and submits a focused fix or source draft. No GitHub account is required.
2. **Queue:** the submission becomes a labeled GitHub issue with page path, review tier, citation fields, and a focused or full-source diff.
3. **Triage:** an editor marks it `accept`, `needs source`, `needs expert`, `reject`, or `duplicate`.
4. **Draft:** accepted changes move into Decap CMS or a pull request.
5. **Checks:** content QA, build, and citation review run before merge.
6. **Publish:** changes do not reach `main` without review.
7. **History:** GitHub commit history remains the public audit trail.

## Protected Pages

Pages or edits involving legal, tax, VAT, RJSC, employment, fintech, healthtech, privacy, import/export, payments, banking, or government policy are treated as protected. Low-risk copyedits can still be reviewed quickly, but factual changes need a source and high-risk changes need expert or senior editor signoff.

## Citation Rules

- Put an inline link next to factual claims, or cite the source clearly in the same section or source section.
- Prefer official portals, gazettes/rules, regulator notices, annual reports, recognized research reports, and primary datasets.
- Blog posts, news, and founder experience can add context, but they do not replace official sources for legal, tax, or regulatory truth.
- Do not cite only a homepage when an exact notice, report, or service page exists.
- Update `last_verified` when the date of checking matters.

## AI Use

AI may help draft, compare, translate, summarize, and create checklists. AI is not a source, reviewer, or approval authority.

Before publishing AI-assisted changes, a human must:

- open the sources and verify the claims
- separately review high-risk content
- check for fabricated citations

## Correction Policy

When a serious error is confirmed:

1. edit or remove the affected claim
2. update the source
3. add a warning note when useful
4. explain the change in the issue or pull request
